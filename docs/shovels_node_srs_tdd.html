Shovels for n8n

A verified, dependency-free n8n integration for the Shovels building-permit and contractor API — published by CI with a provenance attestation, and reviewed by n8n itself.
Doc IDMICT-NODE-003
Version1.0 — Baselined
AuthorJake Morgan
StatusApproved for build

This document specifies and designs n8n-nodes-shovels, a community node that wraps the Shovels REST API — building permits and contractors across 1,800+ U.S. jurisdictions — as a first-class n8n integration. It is built in the declarative style (the node is a routing table n8n's HTTP layer executes, not an imperative script), carries a credential type for the X-API-Key, and exposes a small, well-described set of operations over the Permit, Contractor, and Address resources, with pagination and geo_id resolution handled by configuration. The node's defining properties are the ones the n8n Verified Community Nodes Program demands and rewards: zero runtime dependencies, no filesystem or environment access, an English-only interface, exactly one third-party service, and — required for verification since 1 May 2026 — a release that is published from CI with a cryptographic provenance attestation, never from a laptop. The whole package is held to a structural bar: npx @n8n/scan-community-package must pass, locally and in CI, before any publish. The document is in two parts: Part I states what the node must do and be (SRS); Part II states how it is built, released, and verified (TDD).

The integration is configuration; the release is a proof.

Most "I built an integration" portfolio entries are a script that calls an API and an npm package pushed from a laptop. Two things are wrong with that as a credential. First, the trust surface is the author's whole machine and an opaque dependency tree — you take it on faith that the published bytes match the source. Second, nobody reviewed it. This node refuses that shape on both counts. It is written declaratively: the node is a routing table — resources, operations, parameters, and the requests they map to — that n8n's own HTTP layer executes. There is no execute() body to audit and no arbitrary code path; the integration's blast radius is its configuration. It ships with zero runtime dependencies and touches no files or environment variables, so there is no transitive supply chain to compromise and no ambient authority to abuse — the verification program requires this, and it is also the strongest posture available.

The release is where the real contribution lives. Every published version is built by a GitHub Actions workflow from a specific commit and carries a provenance attestation signed through GitHub's OIDC infrastructure — anyone can cryptographically verify which workflow, repository, and commit produced the artifact on npm. n8n then manually vets the package and marks it verified, after which it installs in one click on n8n Cloud and self-hosted alike, with a shield next to its name. That is the line that matters: "I published a package" becomes "a named workflow built this from this commit, and n8n reviewed it." The trust is a property of the pipeline and the review, not of trusting me.

Software Requirements Specification
What the node must do — and be
Defines scope, actors, and the complete set of functional and non-functional requirements the build in Part II is obligated to satisfy — including the program constraints (zero dependencies, provenance publishing, verification) that distinguish a verified node from a published one.

§1 Introduction

1.1 Purpose

This specification defines the requirements for n8n-nodes-shovels, an n8n community node that makes the Shovels API — building permits, contractors, and the geographies that index them — usable inside an n8n workflow without writing an HTTP-Request node by hand each time. It is the authoritative reference for what the node must do, what it must not do, and the standard it must meet to be accepted into the n8n Verified Community Nodes Program. The motivating context is the author's own: Morgan ICT operates in the low-voltage construction trade, where permit and contractor signals are direct go-to-market intelligence, and an automation engineer who can ship a verified integration for that exact domain demonstrates both the language (TypeScript, n8n internals) and the discipline (supply-chain provenance, external review) that the role rewards. The deliverable is not a demo; it is a reviewed, installable integration carrying a shield in the n8n nodes panel.

1.2 Scope

In scope

  • A declarative-style node scaffolded with the n8n-node CLI — the structure and tooling the verification program expects — exposing a Resource / Operation parameter pair.
  • A credential type for Shovels X-API-Key header authentication, with a credential test request so users get immediate green/red feedback.
  • Three to four operations across Permits and Contractors — search permits by geography, get a permit by id, search contractors, get a contractor by id — plus an Address resolution operation that turns an address into a geo_id.
  • Pagination over the API's page / next_page scheme, and geo_id resolution (including the shortcut that a state or ZIP is already a geo_id), handled in routing/config.
  • Clean operation and parameter descriptions, sensible field grouping (required fields surfaced; optional filters under Additional Fields), and typed, user-legible error surfacing.
  • A thorough README with screenshots, example workflows, and authentication setup; an MIT license; the n8n-community-node-package keyword for discovery.
  • A GitHub Actions publish pipeline that builds, scans, and publishes to npm with a provenance statement, and submission through the n8n Creator Portal for verification.
  • Optional: a polling trigger ("new permits in area X since last run") for the same service, allowed alongside the main node.

Out of scope

  • The full Shovels surface. Addresses-as-data, Markets/metrics, Lists, and the Decisions API beta are deliberately deferred (§21) to keep v1.0 small, well-described, and easy to review — one tight resource set beats a sprawling one.
  • Any runtime dependency. Verified nodes forbid them; the package's dependencies map is empty by requirement, not by accident (§3.7, §16).
  • Filesystem or environment-variable access of any kind; all input arrives through node parameters and the n8n credential store (verification constraint).
  • Logic / flow-control behavior, or anything that competes with an n8n paid/enterprise feature — categories the program does not accept.
  • Caching, rate-limit budgeting, or analytics layered on top of the API; the node is a faithful, thin client, not a platform over Shovels.

1.3 Intended Audience

Hiring reviewers and engineers evaluating the work; the implementer (the author) during the build; n8n's verification reviewers; and any future maintainer adding resources or operations. No prior familiarity with n8n node internals or the Shovels API is assumed — the node interface is specified in §5 and §11, and the release mechanics in §19.

1.4 Definitions, Acronyms & Abbreviations

A full glossary appears in Appendix C. The terms that recur most: declarative node (a node defined as configuration/routing that n8n executes, vs a programmatic execute()), resource / operation (the two-level parameter pattern n8n uses to organize a node's actions), geo_id (Shovels' unified geographic identifier), credential type (the n8n object that holds and injects the API key), provenance (a signed attestation of which workflow/commit built an npm package), and verified node (a community node n8n has reviewed and marked installable from the canvas with a shield).

1.5 References & Standards

  • RFC 2119 — keyword strength (MUST / SHOULD / MAY).
  • The n8n community-node verification guidelines and code standards — the acceptance bar this node targets (§3.7, §13).
  • The Shovels REST API (v2) reference — Permits, Contractors, Addresses, geographies, the X-API-Key header, and the page/next_page pagination model (§5.1).
  • npm package provenance (build provenance via GitHub Actions OIDC / Sigstore) — the attestation mechanism required for verification since 1 May 2026 (§19).
  • SemVer 2.0.0 — version numbering for the published package.

1.6 Document Conventions

Requirements carry stable identifiers of the form FR-XX-n (functional, grouped by concern) and NFR-XX-n (non-functional). Each states a single testable obligation and a strength keyword. Appendix A traces every identifier to the design section that satisfies it.

§2 Overall Description

2.1 Product Perspective

The node has two faces that this document keeps deliberately separate. The runtime artifact is what a user installs and drops on a canvas: a credential and a set of operations that, on execution, issue authenticated HTTP requests to Shovels and return items. It is small, deterministic, and dependency-free. The release artifact is the npm package and its provenance: the bytes, their signature, and the n8n verification status that together let a stranger trust the install. Most of the engineering interest — and the program's bar — lives in the second face. The node is not a service the author operates; it is a published, reviewed component that runs inside other people's n8n instances, which is precisely why supply-chain integrity is the headline rather than uptime.

2.2 Product Functions

At a glance — runtime: authenticate (X-API-Key) → pick resource (Permit / Contractor / Address) → pick operation (search / get / resolve) → set fields (geo_id + date window + filters, or an id) → execute → paginate if requested → emit items. Release: scaffold (n8n-node CLI) → implement (declarative routing) → build (tsc) → scan (@n8n/scan-community-package) → test in self-hosted n8n → tag → CI publish to npm with provenance → submit to the Creator Portal → verified → one-click install. Each verb maps to a component in §10 and a requirement group in §3.

2.3 Actors & User Classes

ActorTypeInteraction

2.4 Operating Environment

At runtime the node loads inside any n8n instance that supports community nodes — self-hosted on the author's machine for development and testing, and (once verified) n8n Cloud and self-hosted for end users, requiring an n8n version that exposes verified-node install from the canvas. The node calls Shovels over outbound HTTPS only. The build/release environment is a CI runner (Ubuntu, a pinned Node LTS) and the public npm registry; the development loop is a local self-hosted n8n with the package linked in.

2.5 Design & Implementation Constraints

  • The package MUST declare zero runtime dependencies (empty dependencies); only devDependencies and a peerDependency on n8n-workflow are permitted. This is the top-level verification constraint (§3.7).
  • The node MUST NOT read or write files, and MUST NOT read environment variables; every input arrives via node parameters or the n8n credential store.
  • The node MUST be built declaratively where the API allows it (routing-based), scaffolded with the n8n-node CLI; an imperative execute() is reserved only for what declarative routing genuinely cannot express (e.g. the optional trigger's poll()).
  • The package MUST integrate exactly one third-party service (Shovels); a trigger node for the same service MAY be included alongside the main node, but nothing else.
  • For verification, the package MUST be published from a GitHub Actions workflow with a provenance statement — never from a local machine (§19).
  • The node interface and all documentation MUST be in English only; the package MUST be MIT-licensed and carry the n8n-community-node-package keyword.
  • Secrets (the API key) MUST live only in the n8n credential store and MUST NOT appear in source, logs, or the published artifact.

2.6 Assumptions & Dependencies

  • The Shovels API remains stable at its v2 surface for the operations wrapped (paths, the X-API-Key header, and the page/next_page response shape). API specifics are confirmed against the live reference at build time, not assumed from memory.
  • n8n's declarative routing supports the request mapping, the credential generic-auth header injection, the credential test request, and the generic pagination this node relies on (§15).
  • The user supplies their own valid Shovels API key; the node neither provisions nor bundles one.
  • npm provenance and the GitHub Actions OIDC signing path are available to the repository (public repo, supported runner) (§19).
  • A state two-letter code (e.g. CA) and a ZIP (e.g. 94103) are themselves valid geo_ids and need no lookup; cities, counties, and specific addresses require resolution via the Address endpoint (§3.4).

§5 External Interface Requirements

5.1 Shovels REST API (outbound)

Base URL https://api.shovels.ai/v2; authentication via the X-API-Key header. The wrapped endpoints: GET /permits/search and GET /contractors/search (require geo_id + permit_from/permit_to; accept filters; return { items:[…], page, size, next_page }); GET /permits/{id} and GET /contractors/{id} (single object by Shovels id); GET /addresses/search (address → geo_id). Pagination is page-based with a next_page cursor in the body; dates are YYYY-MM-DD. Errors are typed by status: 401 (bad/missing key) terminal, 429 (rate limit) retryable with backoff.

5.2 n8n Node Interface (the canvas)

The node renders from its description: a Resource dropdown, an Operation dropdown per resource, required fields, and an Additional Fields collection of optional filters. The credential dialog renders a single masked API-key field with a Test button. Inputs and outputs are the standard main connections; each output item is one permit/contractor/address record (§3.4).

5.3 npm Registry (release / distribution)

The package is published public to npm under n8n-nodes-shovels, carrying the n8n-community-node-package keyword (for discovery), the n8n manifest block (pointing at the compiled node and credential), an MIT license, a repository field, and a build-provenance attestation visible on the package page. This is the artifact n8n fetches for final vetting.

5.4 GitHub Actions (publisher / signer)

A release-triggered workflow builds, scans, and publishes the package, requesting an OIDC token (id-token: write) so npm can record a provenance statement binding the artifact to this repository, commit, and workflow. The scaffolded publish.yml is the basis (§19). This is the only sanctioned path to a verifiable release.

5.5 n8n Creator Portal (verification)

The submission surface for the Verified Community Nodes Program. It accepts a published, provenance-carrying package, runs n8n's manual quality/security review, and — on success — grants the verified shield and canvas-installability across self-hosted and Cloud. Review feedback is handled as an iteration loop (FR-VF-3).

§6 Data Requirements

  • Credential (secret). The Shovels API key — held only in the n8n credential store, encrypted at rest by n8n, injected as a header at request time; never in source, logs, output, or the published package (NFR-SE-1).
  • Transient request/response data. Permit, contractor, and address records flow through the node from Shovels to the workflow; the node neither persists nor caches them. The user's workflow decides what to store.
  • Trigger watermark (optional). If the polling trigger is built, its last-seen permit date/id lives in n8n workflow static data — n8n-managed state, not a file (FR-TR-2).
  • Release metadata. The package version, the tagged commit, the CI run, and the provenance attestation are the durable record of what shipped and how (NFR-OB-1); they live in npm and GitHub, not in the running node.
  • No bundled data. The published artifact contains source, configuration, an icon, and docs only — no datasets, no keys, no client information (Classification).

§9 The Provenance Principle

Declare the contract, depend on nothing, hold no ambient authority, and let the build sign itself and n8n review it.

This piece carries the portfolio's recurring move into a different domain. Pieces I and II each took a non-deterministic step (an extraction, a generation) and wrapped it in a deterministic gate, so the trust came from the structure and not the model. A node has no model to distrust — but it has a worse default failure: a published artifact you take on faith. The discipline here is to make the artifact's trustworthiness a verifiable property rather than a request for faith. Four properties carry it:

  • The node is a contract, not a script. Built declaratively, the node is a routing table — resources, operations, fields, and the requests they map to — that n8n's own HTTP layer runs. There is no imperative body to audit, no place for surprising side effects; the integration's entire behavior is readable as configuration (§10, §15). Its blast radius is its routing.
  • Zero runtime dependencies — the supply chain is the source. The dependencies map is empty. Nothing is pulled at install or run time, so there is no transitive tree to be compromised, no postinstall script to fear, and nothing for a reviewer to chase. The verification program requires this, and it is independently the strongest supply-chain posture a package can take (§16).
  • No ambient authority. The node reads no environment variables and touches no files. The API key lives in n8n's credential store and is injected as the X-API-Key header by the credential's generic-auth config — the node body never sees it. The optional trigger's state lives in n8n-managed workflow static data, not on disk. Every input is an explicit parameter (§16).
  • The release is signed and externally reviewed. Each version is built by a named GitHub Actions workflow from a tagged commit and published with an npm provenance attestation signed via OIDC — anyone can verify which workflow and commit produced the bytes. n8n then manually vets the package and marks it verified. The trust is in the pipeline and the review, not in the author (§19).

This is the same shape as the earlier pieces, relocated from inference to distribution. There, a single risky step was wrapped so that what reaches a user is reproducible and checkable without trusting any one model call. Here, the published artifact is wrapped so that what a stranger installs is reproducible and checkable without trusting the author's laptop: the bytes trace to a commit, the dependency surface is empty, the structure is configuration a linter and a human can read, and an outside party (n8n) has signed off. The capstone (MICT-FIELD-005) inherits exactly this for its own CI publish — Piece III is where the release pipeline earns the word "provenance."

§10 Component Design

Each component lists its responsibility, inputs, outputs, and failure behavior.

10.1 Credential Type — ShovelsApi

Responsibility: hold the API key and inject it as the X-API-Key header on every request; provide a credential test. In: the user's key (masked field). Out: an authenticated request, transparently to every operation. Fails: a bad key surfaces on Test (red) and at runtime as a 401 the node maps to an auth error (§14). The node body never handles the key — least authority by construction (FR-CR-2).

10.2 Node Description — Shovels

Responsibility: declare the node's identity, defaults, credential reference, requestDefaults (base URL + Accept), and the Resource/Operation/field tree with their routing. In: the user's selections on the canvas. Out: the request n8n issues. Fails: nothing to fail imperatively — it is configuration; a malformed description fails the build/scan, not a user run.

10.3 Permit Operations

Search (GET /permits/search): required geo_id + permit_from/permit_to, optional filters, paginated, items unwrapped. Get (GET /permits/{id}): a single permit by id. Both are pure declarative routing (FR-OP-1/2).

10.4 Contractor Operations

Search (GET /contractors/search) and Get (GET /contractors/{id}), mirroring the permit operations' shape so the node is internally consistent and easy to extend (FR-OP-3/4).

10.5 Address Resolution

Resolve geo_id (GET /addresses/search): turns an address into a geo_id for downstream searches. Documents the shortcut that state codes and ZIPs are already geo_ids, so the lookup is only needed for cities, counties, and street addresses (FR-OP-5, FR-PG-4).

10.6 Pagination & Output Shaping

Responsibility: expose Return All / Limit; when returning all, walk next_page via declarative generic pagination until null; unwrap the items array so each record is one n8n item. Fails: a thin or empty page is simply fewer/zero items, not an error; an API error mid-page is surfaced typed (FR-PG-1/2/3, FR-UX-5).

10.7 Error Mapping

Responsibility: turn HTTP failures into n8n node errors carrying status + Shovels message (401 → auth, 429 → rate limit, others → generic API error), and honor continue-on-fail. The declarative layer surfaces non-2xx automatically; this component ensures the message is legible (FR-UX-3/4).

10.8 Polling Trigger — ShovelsTrigger (optional)

Responsibility: on each poll, fetch recent permits for a geo_id, compare against a watermark in workflow static data, emit only new ones, advance the watermark. In: geo_id + poll schedule. Out: new permit items. Fails: an API error skips the cycle without losing the watermark; dedupe prevents replays (FR-TR-1..3). The one place an imperative poll() is warranted — and it still touches no files or env.

§12 Processing Sequences

12.A Request Execution (per operation, at runtime)

  1. Resolve selections. n8n reads the chosen resource/operation and the field values from the node.
  2. Build the request. The operation's routing.request sets method + URL against requestDefaults; send mappings attach query params (geo_id, dates, filters) or the path id.
  3. Authenticate. The credential injects the X-API-Key header (generic auth); the node body never sees the key.
  4. Paginate (search, if Return All). Issue the request; while next_page is non-null, re-issue with page = next_page, accumulating items; else honor Limit.
  5. Shape output. postReceive unwraps items so each record is one n8n item; a single-object Get returns that object.
  6. Handle errors. A non-2xx surfaces as a typed node error (status + message); continue-on-fail, if set, lets good items proceed (§14).

12.B Build (local, S13)

  1. Scaffold with the n8n-node CLI (declarative template) — node, credential, icon, manifest, publish workflow.
  2. Implement the credential (11.2) and the node routing (11.3–11.6); keep dependencies empty; touch no files or env.
  3. Build with tsc to dist/; confirm the compiled paths match the manifest's n8n block.
  4. Link the package into a local self-hosted n8n instance.
  5. Test each operation live against Shovels: credential test, search + pagination, get, address resolution, and the error/empty cases (§18).

12.C Ship — Publish & Provenance (CI, S14)

  1. Scan locally: npx @n8n/scan-community-package n8n-nodes-shovels must pass (structure, no deps, no fs/env, naming).
  2. Tag a release commit (SemVer); create a GitHub release.
  3. CI builds and scans; a scan failure stops the pipeline (no publish) (FR-SC-4).
  4. CI publishes with npm publish --provenance --access public, the workflow holding id-token: write so OIDC signs the attestation (FR-SC-2).
  5. npm records provenance binding the artifact to this repo + commit + workflow; the package page shows it.

12.D Ship — Verification (Creator Portal, S14)

  1. Pre-flight the gating rules (FR-VF-1): new integration, one service, no logic/flow-control, no env/file access, English-only, MIT, scan passing.
  2. Submit the published, provenance-carrying version through the Creator Portal.
  3. n8n vets the package (manual quality/security review).
  4. Iterate on any change-request: fix, publish a new signed version, re-submit (FR-VF-3).
  5. Verified: shield + one-click install on Cloud and self-hosted. "Published a package" becomes "n8n reviewed and verified my integration."

The case study (S14) reports this sequence with the artifacts attached: the npm provenance panel, the CI run, the scan output, and the verification status — the same instinct as the other pieces, that a screenshot of a passing pipeline outweighs a paragraph of claims.

§13 The Verified-Node Discipline

The trustworthiness of the release rests on three gates working together, none of which is the author's word.

The scan is the structural gate. @n8n/scan-community-package runs locally and in CI and enforces the things a reviewer would otherwise check by hand: the package structure, the empty dependency map, the absence of file/env access, naming conventions, and the manifest. A scan failure is a red build — it cannot be waved through (FR-SC-4, NFR-QA-1). This is Piece I's schema gate in a new place: a deterministic check that must pass before anything moves downstream.

Provenance is the release gate. Publishing happens only from CI, only from a tagged commit, only with an OIDC-signed attestation. After 1 May 2026 a hand-published package simply cannot be verified, so the discipline is enforced by the program, not just chosen. The attestation is the proof that the bytes on npm are the bytes the workflow built from that commit — the supply chain is closed and inspectable (FR-SC-2/3/6).

n8n verification is the external review. The final gate is a human at n8n vetting the package for quality and security. Passing it is what converts a published package into a credential: the shield, and one-click install across Cloud and self-hosted. The author cannot grant this to themselves — which is exactly why it is worth having (FR-VF-2).

As with the earlier pieces, the property is structural — it holds because of where the gates sit, not because the author is trusted.

A clean README and a green npm page tell a user the package exists; they do not tell them the published bytes match the source, that no postinstall script runs, or that anyone reviewed it. Provenance answers the first cryptographically: the attestation binds the artifact to a specific workflow and commit, so a tampered or laptop-pushed build is detectable rather than indistinguishable. Verification answers the third: n8n looked. The empty dependency map answers the second: there is nothing to run. The package being installable is the convenience; the provenance and the review are the guarantee — and they are the part that generalizes to the capstone's release pipeline.

§15 Request & Routing Configuration

Because the node is declarative, "how it makes a call" is configuration, not code — which is what keeps the trust surface small and the package reviewable. The mechanics:

  • Request defaults. requestDefaults sets the base URL (…/v2) and Accept: application/json once; every operation inherits them, so no operation hard-codes a host.
  • Operation routing. Each operation's routing.request carries its method and path; path ids use an expression (=/permits/{{ $parameter.permitId }}). One operation maps to exactly one request (FR-OP-6).
  • Parameter send. Search fields declare routing.send with type: 'query' and the API's snake_case property name (geo_id, permit_from, …), so the UI label and the wire name can differ while the mapping stays explicit.
  • Authentication. Injected by the credential's generic-auth header, not by any operation; rotating the auth scheme is a credential change, not a node-wide edit.
  • Pagination. Declarative generic pagination, gated on a Return All boolean, continues while next_page is non-null and advances page from it (§11.6). The user's Limit bounds it when Return All is off, so breadth is always intentional (NFR-PE-1).
  • Output shaping. A postReceive rootProperty unwraps items so downstream nodes receive records, not an envelope (FR-PG-3).
  • Credential test. A bounded request (a state geo_id, a one-day window, size: 1) lets n8n verify the key on save without a meaningful quota cost (FR-CR-3).

§16 Security & Supply-Chain Design

  • Key handling — the API key lives only in n8n's encrypted credential store and is injected as the X-API-Key header by the credential; the node body never reads it, and it never appears in source, logs, the node's output, or the published artifact (NFR-SE-1).
  • No ambient authority — the node reads no environment variables and touches no files; all input is an explicit parameter. There is no path by which the node can exfiltrate local state, because it has no access to any.
  • Zero dependency surface — an empty dependencies map means nothing is fetched at install or run time: no transitive packages, no postinstall scripts, nothing for an attacker to poison upstream or for a reviewer to chase (NFR-DP-1).
  • Provenance as the supply-chain guarantee — every verification-bound version is built by a named CI workflow from a tagged commit and published with an OIDC-signed attestation; the bytes on npm are cryptographically tied to the source. A package pushed from a laptop is both detectable and, post-May-2026, un-verifiable (NFR-SE-2).
  • External review — n8n's verification is an independent security/quality check; the shield is a third party's statement, not the author's.
  • Least privilege at the boundary — the credential is scoped to the single Shovels header; the node integrates exactly one service; the optional trigger's only state is an n8n-managed watermark. Nothing in the package reaches beyond what its one job needs.

§17 Release Observability & Footprint

A node is not a service with uptime to chart; its "observability" is the inspectability of what shipped and how, and its "cost" is the footprint it imposes and the API quota it spends.

What shipped
Every release is answerable from four durable records: the npm provenance panel (workflow + commit), the CI run logs (build + scan + publish), the tagged commit, and a changelog. "What is in x.y.z, built by what, from where" needs no trust to answer (NFR-OB-1).
Install footprint
Source + an icon; zero transitive packages. The node adds negligible weight and negligible runtime overhead beyond the API round-trip (NFR-PE-1).
API quota
The node spends the user's Shovels quota, not the author's. The credential test is deliberately tiny; pagination is bounded by Return All / Limit so a broad query is an explicit choice, not an accident.
Verification status
The shield (or its absence) and the npm provenance badge are the user-facing trust signals, surfaced in the README (FR-DOC-4).

This is the same instinct as the other pieces' eval suites — make the bar a thing that is run, not a thing that is claimed.

§19 CI/CD & the Publish Pipeline

The 2026 differentiator — and the reason this piece is the program's "provenance" exemplar — is that the publish pipeline is itself CI/CD with supply-chain provenance. Releasing is not a command run by hand; it is a workflow that builds, scans, and signs.

  • On a tagged release, a GitHub Actions workflow checks out the commit, installs, builds, and scans.
  • The build fails (no publish) if the scan does not pass (FR-SC-4) — an unscanned or non-compliant package cannot ship.
  • Publishing requests an OIDC token (id-token: write) and runs npm publish --provenance --access public, so npm records an attestation binding the artifact to this workflow, repository, and commit (FR-SC-2).
  • Hand-publishing is forbidden for verification (program rule since 1 May 2026): the workflow is the only sanctioned publisher (FR-SC-3).
  • The README surfaces the provenance and (once granted) verified status, so the trust posture is visible to anyone browsing the package (FR-DOC-4).
publish.yml (sketch)on:
  release:
    types: [published]
permissions:
  contents: read
  id-token: write                       # REQUIRED for OIDC provenance signing (FR-SC-2)
jobs:
  publish:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 22, registry-url: 'https://registry.npmjs.org' }
      - run: npm ci
      - run: npm run build
      - run: npx @n8n/scan-community-package n8n-nodes-shovels   # gate: fail -> no publish
      - run: npm publish --provenance --access public            # OIDC-signed attestation
        env: { NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }} }

This is the line that closes the loop for a reviewer: not "I published an npm package," but "every release is built and scanned in CI and published with a cryptographic provenance attestation, and n8n reviewed and verified the result." It is the operational proof that the guarantees in §9 hold for every version, not just the day it was demoed — and it is the exact discipline the capstone's release pipeline (MICT-FIELD-005, §19) inherits.

§21 Extensibility & Future Work

  • More resources. The declarative pattern makes a new operation a configuration addition: Addresses-as-data, Markets/metrics, Lists (valid tags/filters), and the Decisions API beta are natural next operations, each a routing block alongside the existing ones (NFR-MA-1) — held out of v1.0 to keep the first review small.
  • The polling trigger. The optional "new permits since last run" trigger (§3.9) is the most GTM-flavored thing the package could ship — it turns the node from a query tool into a monitor, and it is the obvious v1.1 once the read operations are verified.
  • Richer field coverage. Shovels' "10x more parameters" (property specs, contractor history, permit details) can be folded into the Additional Fields collections incrementally, without touching existing operations.
  • The capstone tie-in. This verified node plugs into FieldOps Intelligence (MICT-FIELD-005) as an optional permit/jurisdiction enrichment tool for a compliance/takeoff agent — and, more importantly, its CI publish pipeline is the provenance discipline the capstone reuses for its own release. Piece III is where that pipeline was proven.
  • Versioned evolution. A breaking change to the parameter surface becomes node version: 2 behind the same package, so existing workflows keep working — the maintainability path the version field (FR-NS-5) was put there to enable.

B Worked Example — A Query, and a Ship

One real Morgan-ICT-flavored question carried through the node, then the same package carried through the release. (geo_id values and ids are illustrative; the mechanics are exact.)

B.1 — The question

"Which residential HVAC contractors pulled permits in San Jose in the last 90 days?"

B.2 — Resolve the geo_id (San Jose is a city → needs a lookup)

Address - Resolve  -->  GET /addresses/search?...  -->  geo_id: "geo_5f...sj"   # illustrative
# had the question said "CA" or "95112", we'd skip this -- a state/ZIP IS a geo_id (FR-PG-4).

B.3 — Search permits (paginated)

Permit - Search, Return All ON
  geo_id        = geo_5f...sj
  permit_from   = 2026-03-08          # 90 days ago
  permit_to     = 2026-06-06          # today
  property_type = residential          # Additional Fields
  permit_tags   = hvac                 # Additional Fields

response page 1: { items: [ {id:"caf3...", contractor_id:"e79c..."}, ... ], page:1, size:50, next_page:2 }
  --> next_page != null  -->  re-request with page=2  -->  ... until next_page == null
  --> postReceive unwraps items[]  -->  one n8n item per permit (FR-PG-2/3)

B.4 — Get the contractor behind a permit

Contractor - Get, id = "e79c..."  -->  GET /contractors/e79c...
  --> { id:"e79c...", business_name:"...", license:"...", metrics:{...} }   # full object

B.5 — Edge cases (handled, not crashed)

expired key      -->  401  -->  node error "auth: check your Shovels API key" (FR-UX-3)
rate limited     -->  429  -->  retryable / backoff, message names rate limiting
valid, no match  -->  items: []  -->  zero n8n items, NOT an error (FR-UX-5)
city as geo_id   -->  field description steered the user to Resolve first (FR-PG-4/5)

B.6 — The ship walkthrough

npx @n8n/scan-community-package n8n-nodes-shovels   -->  PASS          (FR-SC-4)
git tag v1.0.0 && gh release create v1.0.0          -->  triggers CI
CI: checkout -> npm ci -> build -> scan -> publish --provenance         (FR-SC-2)
npm: n8n-nodes-shovels@1.0.0  +  provenance attestation (repo+commit+workflow)  (AC-9)

C Glossary

Declarative node
A node defined as configuration (a routing table) that n8n's HTTP layer executes, as opposed to a programmatic node with an imperative execute() method. Smaller surface, easier to review.
Resource / Operation
n8n's two-level convention for organizing a node's actions: a Resource (Permit, Contractor, Address) and, per resource, an Operation (Search, Get, Resolve).
Routing
The declarative mapping from an operation/parameter to an HTTP request — method, URL, query/body params, pagination, and output shaping — that n8n runs.
requestDefaults
Node-level defaults (base URL, headers) inherited by every operation, so no operation hard-codes a host.
Credential type
The n8n object that holds a service's secret and injects it into requests. Here, the Shovels API key as an X-API-Key header.
Generic auth
The credential authentication mode for sending auth data in a header, query string, or body — used here to inject the API-key header.
Credential test
A bounded request n8n makes when a user saves a credential, to show a green check or a clear failure.
geo_id
Shovels' unified geographic identifier. A state code or ZIP is itself a geo_id; a city, county, or address must be resolved (via the Address endpoint) first.
Pagination (next_page)
Shovels' page-based scheme: each response carries page, size, and a next_page cursor; iteration ends when next_page is null.
postReceive / rootProperty
A declarative output step that unwraps a nested array (here, items) so each element becomes one n8n item.
Runtime dependency
A package pulled at install/run time. Verified n8n nodes forbid them; this package's dependencies map is empty.
peerDependency
A package the host (n8n) provides at runtime (here n8n-workflow) — declared but not bundled, so it is not a runtime dependency of the node.
Provenance / attestation
A signed statement recording which workflow, repository, and commit built an npm package, letting anyone verify the published bytes' origin.
OIDC (id-token: write)
The GitHub Actions mechanism that lets the workflow obtain a short-lived identity token used to sign the provenance statement.
@n8n/scan-community-package
The linter/scanner that checks a community package against the program's structural rules (no deps, no fs/env, naming, manifest). Must pass for verification.
Creator Portal
n8n's submission surface for the Verified Community Nodes Program.
Verified node / shield
A community node n8n has manually reviewed; it carries a shield and installs in one click from the canvas on self-hosted and Cloud.
n8n-community-node-package
The npm keyword that marks a package as an n8n community node, used for discovery.
Polling trigger
A trigger node that periodically calls an API and emits new results; here, optional, watching for new permits and persisting its watermark in workflow static data.
Workflow static data
n8n-managed per-workflow state a node can read/write across executions — used by the optional trigger instead of a file, keeping the no-fs/no-env constraint.
